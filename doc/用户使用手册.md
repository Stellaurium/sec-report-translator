# SEC Report Translator 用户使用手册

本文面向普通使用者，说明软件功能、命令、配置参数和推荐设置。

## 1. 软件功能

SEC Report Translator 提供四个命令：

1. `init-config`：生成配置文件模板。
2. `download`：从 SEC 下载 filing。
3. `unpack`：拆解 SEC `full-submission.txt`。
4. `translate`：翻译一个 HTML filing 文档。

推荐工作流：

```text
init-config -> download -> unpack -> translate
```

也可以只使用其中某个步骤。例如已经有 `full-submission.txt` 时，可以跳过下载；已经有 HTML 文件时，可以直接翻译。

## 2. 生成配置文件

```powershell
.\.venv\Scripts\sec-translator.exe init-config -o sec-translator.toml
```

如果文件已经存在，命令会报错。确认要覆盖时：

```powershell
.\.venv\Scripts\sec-translator.exe init-config -o sec-translator.toml --overwrite
```

配置文件必须显式提供。程序不会从环境变量读取 API key，也不会用环境变量覆盖配置文件。

## 3. 下载 SEC filing

```powershell
.\.venv\Scripts\sec-translator.exe download --ticker PDD --form 20-F --limit 3 -o downloads --config sec-translator.toml
```

常用参数：

- `--ticker`：股票代码，例如 `PDD`、`AAPL`、`BABA`。
- `--form`：SEC 表格类型，例如 `20-F`、`10-K`、`10-Q`、`8-K`、`6-K`。
- `--limit`：最多下载几份。
- `--after`：只下载该日期之后的 filing，格式 `YYYY-MM-DD`。
- `--before`：只下载该日期之前的 filing，格式 `YYYY-MM-DD`。
- `-o, --output-dir`：下载根目录。
- `--config`：配置文件路径。

示例：下载 2023 年之后最多 2 份 20-F：

```powershell
.\.venv\Scripts\sec-translator.exe download --ticker PDD --form 20-F --after 2023-01-01 --limit 2 -o downloads --config sec-translator.toml
```

下载结果通常位于：

```text
downloads/sec-edgar-filings/<TICKER>/<FORM>/<ACCESSION>/full-submission.txt
```

SEC 下载要求 User-Agent 中包含可识别身份信息，因此必须配置 `[sec]`。

## 4. 拆解 full-submission.txt

```powershell
.\.venv\Scripts\sec-translator.exe unpack path\to\full-submission.txt -o output\PDD-20F
```

输出目录采用扁平结构，所有文件放在同一个目录：

```text
output/PDD-20F/
  full-submission.txt
  manifest.json
  pdd-20231231x20f.htm
  image1.jpg
  ...
```

`manifest.json` 会记录 filing 元信息和每个文件的：

- sequence
- type
- filename
- description
- content_kind
- size
- decode_status

如果输出文件已经存在，默认报错。确认要覆盖时：

```powershell
.\.venv\Scripts\sec-translator.exe unpack path\to\full-submission.txt -o output\PDD-20F --overwrite
```

## 5. 翻译 HTML

```powershell
.\.venv\Scripts\sec-translator.exe translate output\PDD-20F\pdd-20231231x20f.htm --config sec-translator.toml
```

默认输出在输入文件旁边：

```text
pdd-20231231x20f_translated.htm
pdd-20231231x20f_translation_report.json
pdd-20231231x20f_translation_report.md
```

指定输出路径：

```powershell
.\.venv\Scripts\sec-translator.exe translate output\PDD-20F\pdd-20231231x20f.htm -o output\PDD-20F\pdd-20231231x20f_zh.htm --config sec-translator.toml
```

指定 `-o` 后，报告名会跟随输出文件：

```text
pdd-20231231x20f_zh.htm
pdd-20231231x20f_zh_translation_report.json
pdd-20231231x20f_zh_translation_report.md
```

如果输出文件或报告已存在，默认报错。确认要覆盖时：

```powershell
.\.venv\Scripts\sec-translator.exe translate output\PDD-20F\pdd-20231231x20f.htm --config sec-translator.toml --overwrite
```

## 6. 配置文件详解

### 6.1 `[model]`

主模型配置。

```toml
[model]
base_url = "https://api.deepseek.com/v1"
api_key = "YOUR_API_KEY"
model = "deepseek-v4-flash"
temperature = 0.1
top_p = 0.95
timeout_seconds = 300
max_retries = 3
context_window_tokens = 1000000
max_output_tokens = 16000
```

参数说明：

- `base_url`：OpenAI-compatible API base URL，不要包含 `/chat/completions`。
- `api_key`：API key，必须填写。
- `model`：模型名称。
- `temperature`：温度。翻译任务建议 `0` 到 `0.2`。
- `top_p`：采样参数。通常保持默认即可。
- `timeout_seconds`：单个请求超时时间。
- `max_retries`：API 调用失败时的重试次数。
- `context_window_tokens`：模型上下文窗口，用于估算 block 大小。
- `max_output_tokens`：单次请求最大输出 token。

### 6.2 `[fallback_model]`

兜底模型配置。只有 `enabled = true` 时才会使用。

```toml
[fallback_model]
enabled = true
base_url = "http://localhost:1234/v1"
api_key = "lm-studio"
model = "qwen3.5-9b"
temperature = 0.1
top_p = 0.95
timeout_seconds = 300
max_retries = 1
context_window_tokens = 128000
max_output_tokens = 4096
```

使用时机：

- 主模型失败。
- 严格 JSON 重试失败。
- repair 失败。
- mini-block 重试失败。

如果 fallback 成功，译文会被写入 HTML，并添加 `sec-translator-fallback` 标记样式，方便复核。

如果 fallback 未启用或也失败，程序会保留原文，并在报告中记录。

### 6.3 `[sec]`

SEC 下载配置。

```toml
[sec]
user_agent_name = "Your Name or Company"
user_agent_email = "your_email@example.com"
```

SEC 要求自动化下载提供可识别 User-Agent。邮箱必须是合法格式。

### 6.4 `[batch]`

batch 参数。当前默认推荐使用 block 模式，因此这些参数主要用于：

- 非 block 模式。
- block 模式中的稳定增长/缩小策略。
- block 中 `target_batch_units` 的上限约束。

```toml
[batch]
initial_units = 20
min_units = 1
max_units = 80
stable_successes_to_grow = 5
grow_factor = 1.25
shrink_factor = 0.5
max_chars_per_batch = 12000
prev_context_chars = 3000
next_context_chars = 2000
```

参数说明：

- `initial_units`：线性模式初始 batch 单元数。
- `min_units`：最小 batch 单元数。
- `max_units`：最大 batch 单元数。
- `stable_successes_to_grow`：连续成功多少次后尝试增大 batch/block。
- `grow_factor`：增长倍率。
- `shrink_factor`：失败或上下文受限时的缩小倍率。
- `max_chars_per_batch`：线性模式单 batch 粗略字符上限。
- `prev_context_chars`：线性模式前文上下文字数。
- `next_context_chars`：线性模式后文上下文字数。

### 6.5 `[translation]`

翻译策略。

```toml
[translation]
target_language = "Simplified Chinese"
conservative_table_translation = true
fallback_to_source_on_failure = true
skip_numeric_cells = true
preserve_company_names = true
```

参数说明：

- `target_language`：目标语言，默认简体中文。
- `conservative_table_translation`：prompt 中要求表格保守翻译。
- `fallback_to_source_on_failure`：失败时保留原文。当前实现采取保守策略，失败不会写入坏译文。
- `skip_numeric_cells`：跳过纯数字/低价值数字单元。
- `preserve_company_names`：prompt 中要求保留公司名、证券代码、专有缩写等。

### 6.6 `[output]`

输出配置。

```toml
[output]
default_suffix = "_translated"
write_markdown_report = true
write_json_report = true
```

参数说明：

- `default_suffix`：未指定 `-o` 时的输出后缀。
- `write_markdown_report`：是否写 Markdown 报告。
- `write_json_report`：是否写 JSON 报告。

### 6.7 `[cache]`

本地磁盘缓存。

```toml
[cache]
enabled = true
cache_dir = ".sec-translator-cache"
```

参数说明：

- `enabled`：是否启用磁盘缓存。
- `cache_dir`：缓存目录。相对路径会解析到输入 HTML 所在目录。

缓存用于中断恢复和重复运行。缓存会随 HTML 内容、prompt、目标语言、模型名变化而失效。

### 6.8 `[block]`

block 翻译和并发配置。

```toml
[block]
enabled = true
max_input_ratio = 0.12
target_batch_units = 1
max_batch_chars = 80000
max_batches_per_block = 80
max_workers = 50
warmup_first = true
warmup_delay_seconds = 2.0
before_context_ratio = 0.10
block_body_ratio = 0.80
after_context_ratio = 0.10
```

参数说明：

- `enabled`：是否启用 block 模式。建议开启。
- `max_input_ratio`：最多使用模型上下文窗口的比例。实际输入预算还会扣除系统 prompt 和 `max_output_tokens`。
- `target_batch_units`：每个 batch 的翻译单元数。推荐 `1`，稳定且便于验证。
- `max_batch_chars`：单个 batch 的字符上限。
- `max_batches_per_block`：一个 block 最多包含多少 batch。
- `max_workers`：block 内并发请求数。
- `warmup_first`：是否先发送公共 prompt 预热服务端缓存。
- `warmup_delay_seconds`：warmup 后等待秒数。
- `before_context_ratio`：block 输入预算中分配给前文上下文的比例。
- `block_body_ratio`：分配给当前 block batch 内容的比例。
- `after_context_ratio`：分配给后文上下文的比例。

## 7. 推荐配置

### 7.1 稳定优先

```toml
[block]
enabled = true
max_input_ratio = 0.12
target_batch_units = 1
max_batches_per_block = 80
max_workers = 20
warmup_first = true
warmup_delay_seconds = 3.0
before_context_ratio = 0.10
block_body_ratio = 0.80
after_context_ratio = 0.10
```

适合长文件和网络不太稳定的环境。

### 7.2 速度优先

```toml
[block]
enabled = true
max_input_ratio = 0.12
target_batch_units = 1
max_batches_per_block = 80
max_workers = 50
warmup_first = true
warmup_delay_seconds = 2.0
before_context_ratio = 0.10
block_body_ratio = 0.80
after_context_ratio = 0.10
```

适合 provider 并发额度充足、网络稳定、请求超时配置合理的场景。

### 7.3 本地 OpenAI-compatible 服务

```toml
[model]
base_url = "http://localhost:1234/v1"
api_key = "local-key"
model = "local-model"
temperature = 0.1
top_p = 0.95
timeout_seconds = 300
max_retries = 1
context_window_tokens = 128000
max_output_tokens = 4096
```

只要服务兼容 OpenAI chat completions，即可作为主模型或 fallback 模型。

## 8. 报告如何阅读

JSON 和 Markdown 报告会显示：

- `total_units`：翻译单元总数。
- `cache_hits`：本地磁盘缓存命中数。
- `primary_success`：主模型最终成功数，包含 repair 和 mini-block 成功。
- `repair_success`：反思修复成功数。
- `mini_block_success`：局部 mini-block 重试成功数。
- `fallback_success`：fallback 模型成功数。
- `source_retained`：最终保留原文数。
- `failure_reasons`：失败类型统计。
- `usage.primary.prompt_cache_hit_tokens`：服务端 prompt cache 命中 token。
- `usage.primary.prompt_cache_miss_tokens`：服务端 prompt cache 未命中 token。
- `usage.primary.prompt_cache_hit_ratio`：命中比例。

判断费用和缓存效果时，应同时看：

- 本地 `cache_hits`
- 服务端 `prompt_cache_hit_ratio`
- 总 `completion_tokens`

## 9. 常见问题

### 输出文件已存在

默认不会覆盖任何输出或报告：

```text
Output file already exists: ... Add --overwrite to replace it.
```

确认要覆盖时添加：

```powershell
--overwrite
```

### 下载 SEC 返回 403

常见原因：

- `[sec]` 信息无效或过于泛化。
- 网络、代理或 SEC 临时限制。
- 请求过于频繁。

请确认邮箱格式正确，并避免短时间内大量请求。

### 翻译完成后仍有英文

这是预期的保守行为。可能原因：

- 单元是纯编号、表格编码或低价值文本。
- 模型多次失败。
- 数字或占位符验证失败。
- fallback 未启用。

具体条目可查看报告中的 `retained_units`。

### 如何减少费用

建议：

- 启用 `[cache]`。
- 启用 `[block]`。
- 保持 `target_batch_units = 1` 和较大的 `max_batches_per_block`。
- 开启 `warmup_first`。
- 不要频繁修改 prompt、模型名、目标语言或 HTML 源文件。
- 使用报告中的 `prompt_cache_hit_ratio` 验证服务端缓存是否命中。

### 如何减少保留原文

建议：

- 启用 fallback 模型。
- 降低 `max_workers`，避免网络/服务端不稳定。
- 适当增大 `timeout_seconds`。
- 检查报告中的 failure reason，判断是格式问题、数字问题还是模型拒绝。
